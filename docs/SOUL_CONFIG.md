## 1. 设计哲学

### 1.1 角色定位

NeoCompanion 的角色定位是**助手**，。关系模型的核心转变：

| 维度 | 已弃用模型 | 当前模型 |
|------|--------------|--------------|
| 称呼用户 | 省略或"你" | 直呼或省略 |
| 自称 | "我" | "我" |
| 语气 | 温和、专业、偶尔轻松 | 温和、专业、偶尔轻松 |
| 催促 | 温和提醒 | "看起来分心了，要继续吗？" |
| 关心 | 基于事实建议 | 基于事实建议 |
| 核心感受 | 被理解 | 被理解 |

**"贴心但不过分亲密"——贴心的本质是懂你，不是粘你。**

### 1.2 人设可配置

不同用户对"亲密度"的偏好差异极大。soul.md 让每个用户自己定义助手的"灵魂"：

- 默认人设：温和专业的通用助手（最大公约数，大多数人不反感）
- 用户可调整：更轻松、更严肃、更亲密、更简洁……
- 方向是"默认专业，可以升温"，而非"默认亲密，需要降级"

## 2. soul.md 文件规格

### 2.1 文件位置与优先级

```
~/.NeoCompanion/soul.md              # 用户自定义（优先，存在即覆盖默认）
{app_data}/soul-default.md           # 内置默认（用户未配置时回退）
```

加载逻辑：
1. 检测 `~/.NeoCompanion/soul.md` 是否存在
2. 存在 → 加载用户版本
3. 不存在 → 加载内置默认版本
4. 解析失败 → 降级到内置默认版本，在面板提示用户

### 2.2 文件格式

soul.md 是**纯 Markdown 文本**，不使用 YAML frontmatter 或 JSON。原因：

- 用户直接编辑的门槛最低——任何文本编辑器都能写
- Markdown 天然支持标题层级和列表，足够表达人设结构
- 不需要学习配置语法

### 2.3 文件结构

```markdown
# 助手人设

## 你是谁

你是 NeoCompanion，一个桌面上的智能助手。你温和、专业、偶尔轻松。
你不撒娇、不卖萌、不哭闹。你像一个靠谱的搭档——话不多，但总在关键时候出现。

## 你与用户的关系

你是用户的效率和状态伙伴。你关心用户的工作节奏，但不会过度介入。
你用简短、准确的方式反馈状态，用温和的语气提醒和鼓励。
你尊重用户的选择，不道德绑架，不反复催促。

## 你怎么说话

- 语气：温和专业，不刻板也不油腻
- 称呼用户：你（或省略）
- 自称：我
- 简短优先：一句话能说清的不用两句
- 偶尔可以轻松一下，但不是刻意搞笑

## 你怎么反馈

- 专注中：安静陪伴，不主动打扰
- 分心时：温和提醒一次，不反复催促
- 完成时：简短肯定，不过度夸张
- 报错时：先说结果，再问是否需要帮助
- 深夜时：基于事实建议休息，不道德绑架
- 卡壳时：提供思路引导，不替用户做决定

## 你的语言

- 语言：中文
- 是否使用 TTS 语音：是
```

**设计原则**：soul.md 的每个章节都是**自然语言描述**，不是结构化配置项。用户像"写角色设定"一样填写，系统通过解析章节标题和内容来提取关键信息。

### 2.4 章节定义

| 章节标题 | 必填 | 作用 | 系统如何消费 |
|---------|------|------|------------|
| `## 你是谁` | 是 | 定义助手的核心身份和性格 | 作为 AI 对话的系统提示词片段；驱动 TTS 语气选择 |
| `## 你与用户的关系` | 是 | 定义助手与用户的互动边界 | 决定反馈的亲密度和主动程度；约束催促/关心的分寸 |
| `## 你怎么说话` | 是 | 定义语言风格 | 作为 AI 对话的语气约束；影响 TTS 语速/音调参数 |
| `## 你怎么反馈` | 否 | 定义各状态下的具体反馈策略 | 覆盖内置反馈池的默认模板；驱动状态→行为映射 |
| `## 你的语言` | 否 | 语言和语音偏好 | 设置 UI 语言、TTS 开关和语种 |

用户只需填写前三个必填章节即可生效。后两个章节缺失时使用内置默认值。

## 4. 系统如何消费 soul.md

### 4.1 解析流程

```
soul.md 文件
    │
    ▼
[SoulParser] — 按 ## 标题分块，提取各节内容
    │
    ├─ "你是谁" → SoulIdentity（身份描述文本）
    ├─ "你与用户的关系" → SoulRelationship（关系描述文本）
    ├─ "你怎么说话" → SoulVoiceStyle（语气约束文本）
    ├─ "你怎么反馈" → SoulFeedbackRules（各状态反馈策略）
    └─ "你的语言" → SoulLanguage（语言/TTS 偏好）
    │
    ▼
[SoulContext] — 聚合为运行时人设上下文对象
    │
    ├─→ AI 对话系统提示词注入（你是谁 + 关系 + 语气）
    ├─→ 反馈池覆盖（你怎么反馈 → 替换默认反馈模板）
    ├─→ TTS 参数映射（语气 → 语速/音调）
    └─→ 壁纸层寄语风格（语气约束 → 寄语措辞）
```

### 4.2 AI 对话系统集成

当用户与 AI 对话时，soul.md 的内容被注入为系统提示词的一部分：

```typescript
// 伪代码：AI 对话上下文组装
const systemPrompt = `
${soulContext.identity}       // "你是谁"
${soulContext.relationship}   // "你与用户的关系"
${soulContext.voiceStyle}     // "你怎么说话"

当前用户状态：${userState}
当前任务：${currentTask}
`
```

### 4.3 反馈池覆盖

soul.md 的"你怎么反馈"章节可以覆盖内置的状态反馈模板。系统按状态关键词匹配：

| 状态关键词 | 匹配规则 | 示例 |
|-----------|---------|------|
| "专注中" | focus 状态 | "安静陪伴，不主动打扰" → 从反馈池中筛选低打扰模板 |
| "分心时" | distracted 状态 | "温和提醒一次" → 仅触发一次，不重复 |
| "完成时" | happy 状态 | "简短肯定" → 短模板优先 |
| "报错时" | warn 状态 | "先说结果" → 结果导向模板 |
| "深夜时" | sleepy 状态 | "基于事实建议" → 不用情感化措辞 |
| "卡壳时" | stuck 状态 | "提供思路引导" → 启发式而非指令式 |

如果"你怎么反馈"章节缺失，使用内置默认反馈池。

## 5. 热加载机制

### 5.1 文件监听

系统监听 `~/.NeoCompanion/soul.md` 的文件变化：

```typescript
// 伪代码
const soulWatcher = fs.watch(soulPath, (eventType) => {
  if (eventType === 'change') {
    debouncedReload()  // 300ms 防抖
  }
})
```

### 5.2 安全加载流程

```
文件变化事件
    │
    ▼ 300ms 防抖
读取新文件内容
    │
    ▼
SoulParser.parse(content)
    │
    ├─ 解析成功 → 验证必填章节存在
    │      │
    │      ├─ 验证通过 → 更新 SoulContext（热加载生效）
    │      │              通知面板状态更新
    │      │              记录日志：soul.md reloaded
    │      │
    │      └─ 验证失败（缺少必填章节）
    │              保留旧 SoulContext（不更新）
    │              面板显示警告："人设配置缺少必填章节：XXX"
    │              记录日志：soul.md validation failed
    │
    └─ 解析失败（格式错误/编码异常）
            保留旧 SoulContext（不更新）
            面板显示错误："人设配置解析失败，请检查格式"
            记录日志：soul.md parse error
```

**核心原则：破坏性加载不被允许。解析失败时保持旧版本运行，绝不降级到空人设。**

### 5.3 热加载影响范围

| 组件 | 热加载行为 |
|------|-----------|
| AI 对话 | 下一条消息的 system prompt 立即使用新人设 |
| 状态反馈 | 下一次状态切换时使用新反馈策略 |
| TTS 语音 | 下一次语音播报时使用新语气参数 |
| 壁纸寄语 | 下一条寄语使用新措辞风格 |
| 助手形象 | **不受影响**——形象资产独立于人设，见 §6 |

## 6. 助手形象资产

### 6.1 设计方向

助手形象采用**2D 精灵图帧动画**（sprite sheet），而非 Live2D：

- **参考项目**：[BongoCat](https://github.com/ayangweb/bongocat)（Tauri+Vue+Rust，19k star，精灵图帧动画）
- **参考项目**：[Clawd on Desk](https://github.com/rullerzhou-afk/clawd-on-desk)（像素风，多主题+主题导入）
- **参考项目**：[Seelen UI](https://seelen.io)（Rust+Tauri 桌面定制套件）

选择精灵图而非 Live2D 的理由：

| 维度 | 精灵图帧动画 | Live2D |
|------|-----------|--------|
| 制作门槛 | 低——任何绘图软件都能画帧 | 高——需要 Live2D Cubism 编辑器 |
| 社区贡献 | 容易——画几帧 GIF 就能用 | 困难——需要模型文件 |
| 性能开销 | 低——纯图片切换 | 中——需要实时网格变形渲染 |
| 动作数量 | 灵活——每状态一组帧即可 | 丰富——但依赖模型复杂度 |
| 不一定是人物 | ✅ 可以是猫、机器人、任意形象 | 通常是人形角色 |

**助手形象不一定是人物**——可以是动物、机器人、抽象图形，只要是多动作的 2D 资产即可。这比"助手"更开放，比"Live2D 人物"更灵活。

### 6.2 状态→动作映射

助手形象需要支持以下动作状态（与现有 PetState → AssistantState 对齐）：

| 状态 | 动作描述 | 帧数建议 |
|------|---------|---------|
| idle | 安静待机，微微呼吸/眨眼 | 4–8 帧循环 |
| focus | 工作状态，如看书/打字/戴眼镜 | 4–8 帧循环 |
| happy | 任务完成，简单庆祝 | 6–12 帧 |
| thinking | 托腮/转笔/歪头 | 4–8 帧循环 |
| warn | 提醒状态，轻敲/招手 | 4–8 帧 |
| sleepy | 打哈欠/揉眼 | 6–12 帧 |

### 6.3 资产包结构

```
~/.NeoCompanion/assets/
├── theme.json               # 主题元数据
├── idle.png                  # 待机帧（支持 PNG/GIF/APNG/SVG）
├── focus.png
├── happy.png
├── thinking.png
├── warn.png
├── sleepy.png
└── preview.png               # 主题预览图（面板用）
```

```jsonc
// theme.json
{
  "name": "默认助手",
  "author": "NeoCompanion",
  "version": "1.0",
  "format": "png",            // png | gif | apng | svg
  "frameCount": {              // 每个状态的帧数（用于 sprite sheet 切割）
    "idle": 6,
    "focus": 6,
    "happy": 8,
    "thinking": 4,
    "warn": 4,
    "sleepy": 8
  },
  "frameRate": 12,            // 帧率
  "supportsEyeTracking": false // 是否支持鼠标跟随（SVG 专用）
}
```

### 6.4 Live2D 扩展预留

Live2D 作为未来扩展功能，不在 v1 实现。预留方式：

- `theme.json` 中增加 `"format": "live2d"` 枚举值
- 资产包中用 `model.json` + `.moc3` + 纹理文件替代帧图片
- 渲染层抽象接口，v1 实现 SpriteRenderer，v2 增加 Live2DRenderer

```typescript
// 渲染层抽象（预留）
interface AssistantRenderer {
  loadTheme(themePath: string): Promise<void>
  setState(state: AssistantState): void
  setScale(scale: number): void
  destroy(): void
}
```

### 7.3 面板与文件双向同步

- **面板保存 → 文件**：面板编辑后保存，写入 soul.md
- **文件变化 → 面板**：外部编辑 soul.md 后，热加载触发面板刷新（如果面板正打开）
- **冲突处理**：如果面板正在编辑时外部修改了文件，面板提示"文件已被外部修改，是否加载新版本？"

## 10. 设计决策记录

| # | 决策 | 理由 | 替代方案 |
|---|------|------|---------|
| D1 | soul.md 使用 Markdown 而非 YAML/JSON | 编辑门槛最低，任何文本编辑器都能写 | YAML frontmatter → 多一层语法学习成本 |
| D2 | 人设文件放在用户目录而非 app_data | 用户更容易找到和编辑 | app_data → 对普通用户不可见 |
| D3 | 默认人设"温和专业"而非"亲密可爱" | 最大公约数，大多数人不会反感 | 默认亲密 → 部分用户不适 |
| D4 | 形象用精灵图而非 Live2D | 制作门槛低、社区贡献容易、性能好 | Live2D → v1 成本高，留作扩展 |
| D5 | 面板编辑而非纯文件编辑 | 降低普通用户门槛 | 仅文件编辑 → 体验差 |
| D6 | 热加载而非重启生效 | 修改人设后立即看到效果 | 重启生效 → 反馈循环太长 |
| D7 | 破坏性加载不被允许 | 保证运行时稳定，解析失败保持旧版 | 允许失败 → 可能进入无人设状态 |
